## 技术栈

- **后端**: Node.js + Express + Socket.IO
- **前端**: 原生 HTML / CSS / JavaScript（SPA，Hash 路由）
- **状态管理**: 服务端内存 `Map`，客户端 `localStorage`
- **实时通信**: Socket.IO（替代小程序云数据库 `watch`）

## 快速开始

### 本地开发

```bash
npm install
npm start
# 访问 http://localhost:3000
```

### 服务器部署

```bash
git clone https://github.com/w7739977/zhajinhua-web.git
cd zhajinhua-web
npm install
pm2 start server.js --name zhajinhua-web
pm2 save
```

后续更新：

```bash
cd ~/zhajinhua-web && git pull && pm2 restart zhajinhua-web
```

### 部署后测试

访问 `http://你的IP:端口/#/test?key=你的密钥` 运行自动化测试（密钥在 `server.js` 中 `TEST_KEY` 常量配置）。

### 部署排障

| 现象 | 原因 | 解决 |
|------|------|------|
| API 行为与代码不符 | pm2 有重复进程占用端口，新进程启动失败 | `pm2 list` 检查，删除重复进程后重启 |
| `EADDRINUSE :::3000` | 3000 端口被其他进程占用 | `lsof -i:3000` 找到占用进程，停掉后重启 |
| 重启次数异常高（↺ 几十次） | 进程启动即崩溃，反复自动重启 | `pm2 logs --err` 查看错误日志定位原因 |
| `git pull` 后无变化 | 代码已更新但 pm2 未重启 | 必须 `pm2 restart zhajinhua-web` 生效 |
| 部署/重启后页面还像在玩 | 房间数据在内存中，`restart` 后服务端房间已清空 | 全员**刷新页面**，**重新开房**；必要时强制刷新以加载新 `app.js` |

## 邀请链接进房流程（Web）

邀请链接/二维码指向 `/#/room/房间号`。与「大厅输入房间号加入」不同，直链**不会**自动跳过昵称步骤：

1. 打开链接后先 `getRoom` 拉取房间；若当前 `playerId` **不在** `room.players` 中，展示 **「加入房间」** 页（昵称输入 + 确认加入 + 返回大厅）。
2. 点 **确认加入** 调用 `POST /api/joinRoom`，服务端将玩家写入 `players` 后，再进入正常房间界面。
3. 若本局已在进行，加入后为观战，与大厅加入逻辑一致。

**原因**：此前仅 `getRoom` 会误以为已进房，实际未 `joinRoom`，房主侧人数不足无法发牌（≥2 人）。

**Socket**：`initRoomPage` 仍会 `joinSocketRoom` 订阅广播；**真正入桌**以 HTTP `joinRoom` 为准。

`getRoom` 失败（房间不存在等）时跳转大厅，避免空白加载。

## 游戏规则

1. 房主创建房间，其他玩家通过邀请链接、二维码或房间号加入；**点邀请链接进来的玩家须先完成「加入房间」页再入桌**（见上一节）
2. **庄家**点击「发牌」，一键给全员发牌（≥2人），每人一张手牌 + 一张公牌
3. 非庄家玩家选择注码（1 / 2 / 3 杯）
4. 庄家选择开牌方式：
   - **选择开牌**：指定与某些玩家比牌，**未选中的玩家手牌保留到下一轮**
   - **全开**：与所有玩家比牌，全胜则过庄
   - **全开不过庄**：与所有玩家比牌，不过庄
5. 每位玩家的最优牌型 = 公牌 + 手牌 + 最优万能牌，三张组合
6. 牌型大小：豹子 > 同花顺 > 同花 > 顺子 > 对子 > 散牌（特殊：2-3-5 散牌可胜豹子）
7. 结算后**庄家**点击「下一局」，全员自动返回房间
8. **全开**且庄家全胜触发过庄时，下一局开始前会 **重新洗满 52 张牌**（同一庄家任期内非过庄局仍消耗同一副剩余牌组，直至不足再过庄洗牌）
9. 游戏进行中新加入的玩家为**观战**状态，下一局自动参与

## 与小程序版的关系

本项目从微信小程序 `pj_zhajinhua` 完整移植而来，两个项目的分支一一对应：

| 小程序 `pj_zhajinhua` | Web `pj_zhajinhua_web` | 用途 |
|:---:|:---:|:---:|
| `main` | `main` | 正式版，不含测试代码 |
| `test/mock-multiplayer` | `test` | 测试版，含测试面板与模拟玩家 |

### 联动开发原则

1. **分支对应修改**：改了哪个项目的哪个分支，必须同步修改另一个项目的对应分支
2. **先改后合**：功能开发在 test 分支完成验证后，再合并到 main 分支
3. **查表同步**：具体代码映射关系见 [SYNC_MAP.md](./SYNC_MAP.md)，修改前先查此表定位对应文件和函数
4. **test 独有内容不进 main**：测试面板、模拟玩家等调试功能只保留在 test 分支
